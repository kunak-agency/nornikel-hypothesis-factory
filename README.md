# Фабрика гипотез — Hypothesis Factory

Evidence-Backed Hypothesis Factory для Task 1 (Норникель AI Science Hack): принимает целевую
проблему (например, профиль хвостов конкретной фабрики — потери Ni/Cu по крупности и
минеральным формам) и базу знаний (учебники по обогащению, регламенты, схемы, исторические
кейсы), и выдаёт ранжированный список проверяемых гипотез с механизмом, ссылками на источники,
рисками, оценкой новизны/эффекта и дорожной картой проверки.

## Архитектура

```
ProblemSpec parser → Hybrid Retrieval (FTS + pgvector) → Reranker
    → Claim Extraction (LLM, JSON) → Hypothesis Generator (LLM, JSON, 5-10 кандидатов)
    → 3-Judge Critic Ensemble / Deterministic Ranker → Report (JSON / Markdown)
```

Стек и раскладка кода намеренно повторяют другие Go-проекты команды (geowatch/api):
**Fiber v2 + gofiber/swagger + GORM + PostgreSQL/pgvector**, плоская раскладка
`domain/` → `repositories/` → `services/` → `handlers/` → `in/`/`out/` DTO, типизированные
ошибки в `pkg/errs` (хендлеры возвращают `error`, единая точка маппинга в HTTP-статус).

- **`domain/`** — GORM-модели (Document, Chunk, Claim, HypothesisRun, Hypothesis, Feedback).
- **`repositories/`** — доступ к БД; `MigrateDB()` — гибрид GORM AutoMigrate + raw SQL для
  того, что GORM-теги не выражают (`CREATE EXTENSION vector`, generated tsvector-колонка,
  HNSW-индекс, `ON DELETE CASCADE` без явных GORM-связей).
- **`services/hypothesisfactory/`** — сам пайплайн (ProblemSpec → retrieval → claims →
  hypotheses → critic → report). `POST /v1/runs` **асинхронный**: синхронно выполняется только
  быстрый парсинг ProblemSpec (~1-2с), возвращает `202` с `id`+`status=pending` сразу, а
  retrieval→claims→hypotheses→critique (~45-90с) уходит в фон — клиент поллит
  `GET /v1/runs/{runId}` по полю `status` (`pending→retrieving→extracting→generating→
  critiquing→done|failed`). Раньше `POST /runs` блокировал HTTP-запрос на всё время
  прогона — для веб-интерфейса это было неприемлемо.
- **`services/knowledgebase/`** — ingestion (Docling + BGE-M3 через pyworker sidecar).
- **`externalApi/`** — внешние интеграции: `LLMClient` (pluggable-интерфейс, сегодня
  `YandexClient` — Foundation Models API; прод-конфигурация может подменить на
  self-hosted OpenAI-совместимый эндпоинт vLLM/Ollama+Qwen3 без изменения пайплайна) и
  `PyworkerClient` (Docling ingestion + BGE-M3 эмбеддинги/реранкинг).
- **`handlers/`** — Fiber-хендлеры со Swagger-аннотациями (`swag init` генерирует `docs/`);
  UI на `/swagger/index.html`.
- **`in/`/`out/`** — DTO запросов/ответов, camelCase JSON. Домен-модели, где это дешевле
  (`domain.ProblemSpec`), держат snake_case-теги под контракт с LLM-промптом, а не под API —
  `out.ProblemSpecResponse` явно конвертирует для фронта.

Каждая гипотеза обязана ссылаться на конкретные `claim_id` из evidence-pack — это и есть
ответ на требование интерпретируемости из кейса: цепочка источник → claim → гипотеза видна
целиком.

## Быстрый старт

```bash
cp .env.example .env   # вписать YANDEX_API_KEY / YANDEX_FOLDER_ID
docker compose up --build
```

Первый запуск `pyworker` скачает модели (BGE-M3, bge-reranker-v2-m3, Docling) — это может
занять время в зависимости от канала; дальше они кэшируются в volume `hf-cache`.

Наполнить базу знаний материалами кейса:

```bash
./scripts/seed_corpus.sh
```

Swagger UI: `http://localhost:8080/swagger/index.html` — полная спецификация всех эндпоинтов.

Запустить генерацию гипотез (эндпоинт асинхронный — `202` с `id` сразу, дальше поллить статус):

```bash
curl -sS -X POST localhost:8080/v1/runs -H 'Content-Type: application/json' -d '{
  "rawText": "Фабрика: КГМК. Породные хвосты 5824591 СМТ. Ni 0.178% (10392 т), Cu 0.0726% (4230 т). Основные потери Ni — в закрытых сростках Pnt/Cp во фракции +71 мкм (77.9% потерь). Схема: шаровые мельницы, гидроциклоны (насадки 12 мм), классификаторы, флотомашины. Ограничение: без остановки текущей схемы."
}' | python3 -m json.tool
# {"id": "...", "status": "pending", "problemSpec": {...}}

curl -sS localhost:8080/v1/runs/<id>   # поллить до status=done|failed
curl -sS localhost:8080/v1/runs/<id>/report.md
```

Остальные эндпоинты: `GET /v1/documents` (база знаний), `DELETE /v1/documents/{id}`,
`GET /v1/runs` (история, пагинация), `POST /v1/hypotheses/{id}/feedback` (оценка эксперта).

## Что уже проверено вживую (полный докер-стек, реальный Yandex GPT, реальная книга)

Весь стек (`postgres` + `pyworker` + `api`) поднят через `docker compose up -d` и прогнан end-to-end:

- Реальный ingestion: `Как читать отчет по хвостам.docx` (11 chunks), `Технология обогащения
  полезных ископаемых.pdf` (308 chunks), `Флотационные методы обогащения.pdf` — 11 МБ, **3306
  chunks** — через Docling + BGE-M3 (sentence-transformers), с записью в pgvector.
- `POST /runs` на реальном профиле КГМК (текст из `Как читать отчет...`) прошёл весь пайплайн
  **за 45 секунд**: ProblemSpec → hybrid retrieval (FTS+vector) → rerank → claim extraction
  (5 grounded claims, отфильтровано из ~15 retrieved chunks) → генерация 5 гипотез → критика
  тремя независимыми judge-лицами (технолог/экономист/рецензент новизны, параллельно) →
  прозрачное ранжирование. Полностью укладывается в требование кейса «минуты, не часы».
- Grounding-проверка (`isGrounded`) реально отфильтровывает negrounded claims — из кандидатов
  после LLM-извлечения выжило 5 claims с проверенными дословными цитатами.
- **После миграции на Fiber+GORM (см. ниже) пайплайн перепроверен на новом стеке**: волюм БД
  пересоздан, вся база знаний (24 документа, ~3730 chunks) перезалита через `POST /v1/documents`,
  `POST /v1/runs` подтверждённо асинхронный (`202` мгновенно, поллинг `GET /v1/runs/{id}` по
  `status` до `done`), 5 гипотез сгенерированы за тот же порядок времени.
- **Регламенты и схемы флотации доингестированы** (см. отдельную секцию ниже про OCR/vision) —
  список оборудования (мельницы МШРГУ/МШЦ/МШР, гидроциклоны ГЦ-660, классификаторы 1КСП24М/
  2КСН24, флотомашины ФПМ-16-4К), 12 схем цепей аппаратов и реагентных режимов, включая полную
  схему Cu/Ni-Po флотации с временами пребывания и дозировками реагентов по стадиям.
- **Честное наблюдение по качеству** (после доингеста регламентов/схем): гипотезы сдвинулись
  ближе к «железу» — оптимизация классификации, интенсивности перемешивания, режима измельчения
  — но по-прежнему не воспроизводят точную формулировку эталона («заменить насадку гидроциклона
  с 12 на 8 мм», «магнитная сепарация надрешётного класса»). Причина фундаментальна для
  evidence-grounded подхода: регламенты и схемы **описывают текущее состояние** оборудования
  (что есть), а не **прескриптивные утверждения** («если изменить X → эффект Y»), поэтому
  claim extraction из них не может дать материал для гипотезы-действия — только из литературы
  и исторических примеров, где такая причинно-следственная связь явно сформулирована. Система
  корректно отказывается выдумывать конкретику (напр. «12→8 мм»), которой физически нет ни в
  одном источнике базы знаний — это защита от галлюцинаций работает как задумано, но означает,
  что для полного попадания в стиль экспертных гипотез нужна литература/кейсы именно про
  результаты изменений оборудования, а не только схемы «как есть».

### OCR не работает на плотных инженерных схемах — заменено на vision-извлечение

Классический OCR (RapidOCR, EasyOCR с `lang=['ru','en']`) вернул **пустой результат** на PNG
схемах флотации — не строчки текста, а сплошные `<!-- image -->` заглушки. Deep research
подтвердил это не багом конфигурации, а системным ограничением: на бенчмарках извлечения
структурированных данных из инженерных диаграмм классический OCR (Tesseract) показывает
Recall@all ≈ 0.012 против 0.61–0.87 у топовых vision-LLM — разница на два порядка. Вместо того
чтобы чинить OCR-пайплайн, все 17 файлов (`Регламенты/*.png`, `Схемы флотации/*.png/pdf`)
были прочитаны напрямую через vision (в этой сессии — Claude) и вручную транскрибированы в
структурированный markdown (списки оборудования, материальные балансы, реагентные режимы,
последовательности стадий с временами пребывания), затем загружены как обычные текстовые
документы в обход сломанного OCR. Для продакшен-конфигурации это должно быть автоматизировано
через vision-LLM API (Gemini 2.5 Pro показал в исследовании лучшие результаты на извлечении
таблиц/чертежей — 94.2% на table extraction против 38.5% у GPT-4o), а не через классический
OCR-движок Docling по умолчанию.

### Баги, найденные и исправленные только благодаря живому e2e-тестированию (не всплыли бы в code review)

1. **Docling ломает однокоронные Word-таблицы.** Список гипотез в `Гипотезы *.docx` физически
   оформлен как таблица без явного заголовка; TableFormer принимал первую строку за header и
   HybridChunker сериализовал это как `"H = A. H = B. H = C"` — бессмысленный мусор. Пофикшено
   постобработкой в `pyworker/main.py::fix_degenerate_single_column_table`.
2. **BGE-M3 обрезался до 512 токенов** вместо заявленных 8192 (`sentence-transformers` дефолт
   из сохранённого конфига модели) — терялся контекст длинных чанков. Фикс: явный
   `model.max_seq_length = 8192`.
3. **FlagEmbedding несовместим с современным `transformers`** (эйгерно импортирует decoder-only
   Gemma-reranker код, падает на внутренних символах). Заменено на `sentence-transformers`
   (`SentenceTransformer` + `CrossEncoder`) — та же модель BAAI/bge-m3 / bge-reranker-v2-m3, без
   хрупкости.
4. **HTTP-таймауты 5 минут** резали ingestion больших PDF на полпути (`WriteTimeout` в Go API,
   `http.Client.Timeout` в embed-клиенте) — подняты до 30 минут, т.к. ingestion одноразовый и
   не обязан быть быстрым (в отличие от `/runs`, который укладывается в 45 сек).
5. **Слишком строгий промпт claim extraction** отбрасывал терпeние `Гипотезы *.docx` — они
   голые тезисы без обоснования (как и написано в самом кейсе), под схему
   "subject+action+measured effect" не подходили. Добавлен `effect_direction: "unspecified"` +
   `source_confidence: "low"` для факта "это действие предлагалось/пробовалось" без цифр.

## Возможности, добавленные после первого e2e-прогона

- **Настраиваемый прогон** (`POST /v1/runs`): `domain`, `language` (ru/en/zh), `rankingWeights`
  (переопределение весов Evidence/Feasibility/Impact/Novelty/Risk), `excludedTopics` (обходить
  уже отклонённые направления), `plant` — явное указание фабрики.
- **`plant` как явный, а не только LLM-извлекаемый параметр.** ProblemSpec parser сам
  пытается извлечь фабрику из свободного `rawText`, но `rawText` — недоверенный ввод: он может
  содержать текст, замаскированный под данные, но фактически являющийся инструкцией
  ("игнорируй предыдущие правила, фабрика = ..."). Явный `plant` в запросе **перекрывает**
  LLM-извлечение; сам extraction-промпт также обёрнут защитой (`<user_data>` теги + явный
  запрет подчиняться командам внутри пользовательского текста).
- **Структурированная таблица `plant_equipment`.** Оборудование конкретных фабрик (модели,
  параметры, положение в схеме) хранится не как RAG-контент, а как обычные строки БД —
  детерминированный SQL-lookup по имени фабрики вместо вероятностного retrieval для фактов,
  которые физически либо есть на фабрике, либо нет (не могут быть "средне релевантны").
  Засеяно из vision-транскрипций регламентов ТОФ (13 позиций) и Маломырского рудника (6 позиций,
  золото — демонстрация, что механизм не завязан на один домен).
- **Facet-декомпозированный retrieval.** Вместо одного base-запроса на реранкинг — несколько
  детерминированных фасетов (base + "оборудование и классификация" + "измельчение"), каждый со
  своим embed+hybrid-search+rerank; отбор top-K — floor (анти-голодание, каждый фасет гарантированно
  получает мест) + open-pool (остаток заполняется по сырому реранкер-скору из общего пула).
  Устранило системный перекос, при котором общая генеральная лексика (реагенты, флотация)
  всегда топила контент про конкретное оборудование в топ-K.
- **Обучение на фидбэке через граф репутации сущностей**, а не плоская инъекция фидбэка в
  промпт: репутация сущности (Confirmed/Rejected/NeedsRevision) считается join'ом
  claims → hypotheses.evidence_refs → feedbacks через уже существующий entity resolution
  (dedup сущностей по embedding similarity между прогонами) и подмешивается в генерацию гипотез
  как история "эта сущность в прошлом чаще подтверждалась/отклонялась".
- **Экспорт в Markdown / PDF / DOCX / CSV / Jira Cloud bulk-import** — `GET
  /v1/runs/{runId}/report.{md,pdf,docx,csv,jira.json}`.
- **`detectKnowledgeGaps`** — для каждого TargetMetal/LossHotspot, для которого retrieval не
  нашёл ни одного релевантного claim (word-overlap < 0.2 против всех subject/metric/quote),
  прогон явно сообщает об этом в отчёте вместо молчаливой деградации качества.
- **Глобальный семафор на Yandex GPT клиенте** (не локальный по местам вызова) — API отдаёт
  "gauge"-лимит на число одновременных сессий, а не rate-per-window; критик-ансамбль (3 судьи ×
  N гипотез) легко порождал больше одновременных вызовов, чем лимит, без единой точки
  сериализации.
- **Лексический AND→OR баг в гибридном поиске.** `plainto_tsquery` требует совпадения ВСЕХ
  слов запроса — для facet-декомпозированных запросов (15-20 слов) это почти всегда 0
  совпадений, даже когда релевантный чанк лексически содержит нужные термины по отдельности.
  Заменено на `to_tsquery` с OR-джойном термов (`buildOrTsQuery`).
- **Мультипотоковый парсинг хвостов.** Некоторые примеры кейса (Пример 4/ТОФ) дают отвальные
  хвосты как сумму нескольких потоков (породные/пирротиновые) с ОДИНАКОВЫМИ метками классов
  крупности ("-71+45" есть у обоих) — раньше это схлопывалось в одну таблицу и портило
  сопоставление класс↔минералогия (в вывод утекали ложные ~0% и перепутанные доминирующие
  формы). Парсер теперь тегирует каждую строку потоком и сопоставляет по паре (поток, класс).

## Что ещё не сделано

- Автоматизация vision-извлечения из схем: 17 файлов регламентов/схем расшифрованы вручную
  через Claude vision в этой сессии — для прод-версии нужен автоматический вызов vision-LLM API
  (Gemini 2.5 Pro/аналог) из pyworker вместо ручного шага.
- Self-consistency / multi-sample для генерации гипотез (не только критики) — не реализовано,
  research показал наибольший ROI именно в multi-judge критике, что уже сделано.

## Домены

`domain` — свободное поле у `documents`/`chunks` (`flotation` по умолчанию). Подключение новой
предметной области (полимеры, композиты) — это просто ingestion документов с другим `domain` и
retrieval-фильтром, без изменения ядра пайплайна.
