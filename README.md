# 🏭 Фабрика гипотез — Hypothesis Factory

**Evidence-backed RAG-система, которая по профилю хвостов фабрики выдаёт ранжированные проверяемые технологические гипотезы — со ссылками на конкретные факты из литературы.**

`Go · Fiber · GORM` · `PostgreSQL / pgvector` · `Python · FastAPI` · `BGE-M3 + bge-reranker-v2-m3` · `YandexGPT`

## 🎯 Что это

**Вход** — описание технологической проблемы фабрики: профиль хвостов (потери Ni/Cu/Au/Ag по
крупности и минеральным формам), схема аппаратов, ограничения площадки.

**Выход** — ранжированный список из 5–10 проверяемых гипотез. Каждая — с механизмом, ссылками
на конкретные факты, рисками, оценкой новизны/эффекта и дорожной картой проверки.

**Ключевой принцип — grounding-first.** Факт не существует для системы, пока не подтверждён
дословной цитатой из источника. Проверка идёт на уровне отдельного утверждения и выполняется
**кодом, а не языковой моделью**. Как следствие, система структурно не может выдумать цифру
(например, «диаметр насадки гидроциклона 12 → 8 мм»), которой нет ни в одном документе корпуса.
Виден весь путь: **источник → факт → гипотеза → оценка**.

## 🚀 Как запустить

Runtime поднимается на **CPU любой машины** одной командой — GPU нужен только для offline-наполнения корпуса.

```bash
cp .env.example .env          # вписать YANDEX_API_KEY / YANDEX_FOLDER_ID
docker compose up --build
```

**Дамп корпуса не хранится в git.** Положите `hypothesis_factory.dump.sql.gz` (~230 МБ) в
каталог `db/` **до первого запуска** — postgres развернёт его автоматически на пустом volume.
Без дампа стек тоже поднимется, просто с пустой базой (см. [`db/README.md`](db/README.md)).

Первый запуск `pyworker` скачает модели BGE-M3 и bge-reranker-v2-m3 (кэшируются в volume).

| Сервис | Порт |
|--------|------|
| api (Go) | `8080` |
| pyworker (Python) | `8090` |
| postgres (pgvector) | `5435` → 5432 |

**Запуск генерации** — эндпоинт асинхронный: сразу возвращает `202` + `id`, дальше поллим статус.

```bash
# 1. Запустить прогон
curl -sS -X POST localhost:8080/v1/runs -H 'Content-Type: application/json' -d '{
  "rawText": "Фабрика: КГМК. Ni 0.178%, Cu 0.0726%. Основные потери Ni — в закрытых сростках Pnt/Cp во фракции +71 мкм (77.9% потерь). Схема: шаровые мельницы, гидроциклоны (насадки 12 мм), флотомашины. Ограничение: без остановки схемы."
}' | python3 -m json.tool
# → {"id": "...", "status": "pending", "problemSpec": {...}}

# 2. Поллить статус до done|failed
curl -sS localhost:8080/v1/runs/<id>

# 3. Забрать отчёт
curl -sS localhost:8080/v1/runs/<id>/report.md
```

Swagger UI: `http://localhost:8080/swagger/index.html`.

## 🧱 Архитектура

Три сервиса, одна команда `docker compose up`:

| Сервис | Стек | Роль |
|--------|------|------|
| **api** | Go · Fiber v2 · GORM | Оркестрация пайплайна, REST API, отчёты (5 форматов) |
| **pyworker** | Python · FastAPI | Эмбеддинги BGE-M3 (мультиязычные, до 8192 токенов) + реранкинг bge-reranker-v2-m3 |
| **postgres** | `pgvector/pgvector:pg16` | Единое хранилище: `vector(1024)` + реляционка + полнотекстовый `tsvector` + HNSW-индекс |

Векторный поиск, реляционные данные и полнотекстовый поиск живут в **одной PostgreSQL** — без
отдельного vector store и без синхронизации хранилищ. LLM подключён через интерфейс `LLMClient`
и заменяется без правки ядра (YandexGPT → self-hosted vLLM / Ollama + Qwen3).

**Пайплайн — 7 стадий, асинхронный.** `POST /v1/runs` парсит проблему и отдаёт `202` за 1–2 с;
остальное считается в фоне (~3 мин на CPU), клиент опрашивает `GET /v1/runs/:id`:

```
pending → retrieving → extracting → generating → critiquing → done
                                                              ↘ failed
```

| Стадия | Что происходит |
|--------|----------------|
| `retrieving` | Фасетный гибридный поиск (FTS + pgvector) + реранкинг |
| `extracting` | Извлечение фактов с детерминированной проверкой обоснования |
| `generating` | Генерация 5–10 гипотез со ссылками на факты |
| `critiquing` | Ансамбль из 3 LLM-судей + воспроизводимое ранжирование |
| `done` / `failed` | Готовый отчёт или явная ошибка |

## 🔬 Почему это работает

Три инженерных решения отличают систему от стандартного «GPT поверх RAG»:

1. **Grounding-first.** На каждый факт — узкий LLM-вызов с дословной цитатой. Обоснование
   проверяет код (`isGrounded`: точное вхождение цитаты **или** пословное перекрытие ≥ 0.6),
   а не модель. Не прошедший проверку факт **никогда не попадает в базу** — это структурная,
   а не декларативная защита от галлюцинаций.

2. **Детерминированные границы доверия.** «Какой аппарат стоит на фабрике» — не может быть
   «средне релевантным»: он либо есть, либо нет. Поэтому оборудование берётся SQL-запросом из
   таблицы `plant_equipment`, а не векторным поиском. Ранжирование гипотез — воспроизводимая
   взвешенная сумма оценок судей, а не ещё один вызов LLM: одни веса → один порядок.

3. **Прослеживаемость.** Весь путь **источник → факт → гипотеза → оценка** прозрачен и
   воспроизводим. Если для целевого металла не нашлось ни одного факта, прогон честно помечает
   пробел в отчёте вместо тихой деградации качества.

ранжированный отчёт. Эксперт оценивает гипотезы (`confirmed` / `rejected` / `needs_revision`),
настраивает веса ранжирования и пересортировывает без повторного вызова LLM. Оператор базы
знаний пополняет корпус через `POST /v1/documents`. Администратор разворачивает стек
(`docker compose up`) и загружает дамп корпуса. Интегратор работает через REST API `/v1`
(асинхронная модель: создать прогон → опрашивать статус → забрать отчёт).

## ⚠️ Ограничения

Честно и коротко:

- **Vision-извлечение схем/регламентов сделано вручную.** Для продакшена нужен автоматический
  вызов vision-LLM из pyworker; шум OCR остаётся источником ошибок.
- **Нет аутентификации** — это прототип для хакатона, не production-развёртывание.
- **Grounding-first по дизайну не даёт предписаний «X → Y»** там, где в литературе нет данных об
  эффекте изменения. Это осознанный компромисс в пользу отсутствия выдумок, а не недоработка.
